Futuro de JavaScript
  • Esto marca el renacer de JavaScript: JavaScript es un lenguaje serio
  • Dinámico
  • Débilmente tipado
  • Orientado a prototipos, no a clases
  • Funciones como elementos de primera-clase
  • closures
  • Soporta programación orientada al objeto, imperativa y funcional
  • Ha salido de la web: aplicaciones móviles, aplicaciones de Windows 8, servidores de aplicaciones...
Ejemplos de tecnologías
  • Zepto.js: sustituto de jQuery para aplicaciones móviles
  • Node.js: servidor de aplicaciones en javascript, código ejecutado EN SERVIDOR
  • CoffeScript: lenguaje que compila a javascript, sintaxis similar a Ruby
  • Backbone.js: framework de estructura (estilo MVC)
  • Meteor: framework END to END
  • Underscore.js: librería para cubrir limitaciones de JS
  • D3.js: librería de SVG
  • Socket.io: gestión de websockets
  • Handlebar: templates/plantillas en JS
Historia
  • 1999: la W3C (la organización de estándares de Internet) publicó HTML4.
  • 1999: la W3C centra todos sus esfuerzos en xhtml
  • 2004: tras una votación en contra de seguir desarrollando HTML, se forma WHATWG, un grupo de gente interesada en evolucionar HTML
  • 2007: la W3C cede y se reintegra en HTML5
  • 2012: Working Draft pero "living standard"

Cache y manifest
  • Caché local en el navegador
  • Soportado por Firefox, Chrome, Safari, IOS y Android 2.0+
  • Muy útil para utilizar en aplicaciones para móviles
  • Fichero Manifest indicamos los ficheros necesarios por el lado del cliente. Cache implicito

  • CACHE MANIFEST
    # v = 1.0.0
    /style.css
    /javascripts/notes.js
    /javascripts/jquery.min.js

  • #v = 1.0.0 -> Indica versión. Al cambiar actualiza ficheros.
  • <html manifest="xxx.appcache">

Web workers
  • JavaScript es mono hilo
  • Permite tareas en background en cliente
  • Cualquier JS puede ser un worker pero no puede tocar DOM
  • Flujo de trabajo:
    • Creación de worker
    • postMessage -> Worker
    • Worker ejecuta la tarea y al termina de nuevo postMessage
    • onMessage cambiamos el DOM
  • Prueba
Web sockets
  • HTTP no permite conexión permanente
  • Web sockets permite conexión a un servidor
  • Eventos:
    • onOpen()
    • onMessage()
    • onClose()
  • Parte cliente en un JS
  • Dos servidores: web y websockets
  • Prueba
